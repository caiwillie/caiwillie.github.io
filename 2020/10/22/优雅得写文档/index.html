标记语言介绍

Baas Doc中的文档应该使用轻量标记语言编写，但不限制使用哪一种标记语言。使用轻量标记语言的最大的好处，是文档编写者不需要掌握文档的排版技巧， 各种文档格式的编辑技术，只需要关注文档本身的内容创作。

Markdown

目前使用最广的文档语言是Markdown，Markdown最大的好处就是语法简单，各大文档平台，笔记软件支持程度好。 IntelliJ IDEA 已经集成 Markdown 工具链，无需安装。

Markdown语法参考

Asciidoc

上面提到Markdown最大的有点是语法简单，但这同样是限制它最大的缺点，一旦你需要一些更复杂的功能，比如表格、交叉引用、脚注、嵌入视频等， 就必须要嵌入原生的HTML或者使用文档平台或者笔记软件提供的”变种“语法，这会导致文档的内容和格式耦合在一起，导致通用性变差。

AsciiDoc语法比Markdown更简洁, 但同时，AsciiDoc提供了强大的功能和灵活性，原生就支持表格、描述列表、警告(提示、注释、警告等)和目录等功能， 不需要嵌入HTML或者使用某种特定的”变种“语法。

除了语法简单外，学习AsciiDoc长远的有点是，他的语法在设计之初就将扩展性考虑在内，这意味了AsciiDoc的功能将会得到不断扩充并丰富， 但编写者不需要学习新的语法或者借助其他工具。

AsciiDoc工具链还未被集成，需要手动安装。

  1. 编辑工具：IntelliJ IDEA 安装 AsciiDoc 编辑插件 或者 下载 AsciidocFX 编辑器

  2. Docker镜像（用于生成文档）：docker pull hub.uban360.com/basic/asciidoc-toolchain

    生成html例子

     docker run -it --rm \
 -v ${PWD}/src:/documents/src \
 -v ${PWD}/output:/documents/output \
 hub.uban360.com/basic/asciidoc-toolchain \
 asciidoctor -r asciidoctor-diagram \
             -o output/index.html src/index-html.adoc

    生成pdf例子

     docker run -it --rm \
 -v ${PWD}/src:/documents/src \
 -v ${PWD}/output:/documents/output \
 hub.uban360.com/basic/asciidoc-toolchain \
 asciidoctor -b pdf -r asciidoctor-pdf -r asciidoctor-diagram \
             -a scripts=cjk -a pdf-theme=theme/cn-theme.yml -a pdf-fontsdir=fonts \
             -o output/index.pdf src/index-html.adoc

文档工程结构

Baas Doc中的文档项目的工程结构需要符合一定的规范，一个标准的的工程结构如下所示：

├── porject_1 // 工程项目1
│   ├── generate-html.sh // 生成html格式文档的脚本，名称固定
│   ├── generate-pdf.sh // 生成pdf格式文档的脚本，名称固定
│   ├── output // 输出目录
│   │   ├── index.html // 生成的html格式文档，名称固定
│   │   └── index.pdf // 生成的pdf格式文档，名称固定
│   └── src // 源码目录
│       ├── xxx_1(.md|.adoc)
│       ├── xxx_2(.md|.adoc)
└── porject_2 // 工程项目2

文件名不得含有空格，不能出现中文，只能出现字母、数字、半角的下划线和连词线。

目录体系

软件用户手册是一部完整的书，建议采用下面的结构。

  • 简介引言（Introduction）：提供对产品和文档本身的概述说明。
  • 快速入门（Getting Started）：引导用户如何快速的使用软件。
  • 安装配置（Install and setup）：如何安装并运行软件。
  • 功能介绍1：主体部分，详细介绍软件的功能
  • 功能介绍2：主体部分，详细介绍软件的功能
  • API参考（API Reference）：软件提供的API文档说明
  • 常见问题（FAQ）：常见问题解答
  • 附录（Appendix）：对文档阅读理解有帮助的其他内容
    • 术语说明（Glossary）：说明文档中出现的特殊术语的含义
    • 最佳实践（Recipes）： 最佳实践介绍
    • 故障处理（Troubleshooting）：故障处理说明
    • 参考资料（Reference）：该文档引用参考的资料
    • 版本说明（ChangeLog）：版本说明
    • 联系方式（Feedback）：联系方式

写作规范

下面规范内容大部分仅为建议或者推荐，并非强制。

句子

  • 句子用词应该清晰明确
    • 尽量使用具体的动词，避免使用含义很泛的动词 
        错误：系统有错误消息
  正确：系统会产生错误消息
    • 副词尽量减少，用具体的量词替代。 
        错误：优化后，系统会运行非常的快。
  正确：优化后，系统会快200%-400%。
    • 用对“的”、“地”、“得”。 ``` 她露出了开心的笑容。 （形容词 ＋ 的 ＋ 名词）

    她开心地笑了。 （副词 ＋ 地 ＋ 动词）

    她笑得很开心。 （动词 ＋ 得 ＋ 副词）

      * 使用代词时避免指代不清

    错误：Python 是解释型语言，但 C++ 是编译型语言，因此它运行效率更高。

    正确：Python 是解释型语言，但 C++ 是编译型语言，因此 C++ 运行效率更高。 ```

  • 句子结构尽量简单
    • 避免使用被动语态 
        错误：假如软件尚未被安装。
  正确：假如未安装软件。
    • 尽量使用肯定句，避免使用否定句。 
        错误：请确认没有接通电源。
  正确：请确认关闭电源。
    • 避免使用双重否定句。 
        错误：用户没有删除权限，不能删除文件。
  正确：用户拥有删除权限，才能删除文件。
    • 名词前不要使用过多的形容词 
        错误：必须在接受过本公司举办的设备培训的技师的指导下使用此设备
  正确：必须在技师的指导下使用此设备，且指导技师需接受过由本公司举办的培训。
    • 避免使用长句子 
        错误：本产品适用于从由一台服务器进行动作控制的单一节点结构到由多台服务器进行动作控制的并行处理程序结构等多种体系结构。
  正确：本产品适用于多种体系结构。无论是由一台服务器，还是由多台服务器进行动作控制，均可以使用本产品。
    • 尽量使用简单句和并列句，避免使用复合句 
        错误：那个昨天生病的人没有参加会议。
  正确：他昨天生病了，没有参加会议。

段落

  • 段落的开头非常重要，需要概述章节的中心思想。
  • 一个段落只能介绍一个主题。
  • 段落内容避免过多或者过少，超过7个句子的章节应该拆分，少于2个句子的章节应该合并。
  • 好的章节应该能回答下面这些问题：
    • 你想让读者知道什么？
    • 为什么读者需要知道？
    • 你该如何让读者知道？

文章

  • 明确文档的描述范围，即会包含什么内容和不包含什么内容。
  • 明确文档的受众，明确文档的阅读者需要具备哪些基础知识。
  • 将主要观点放在文档前面，精心编写文档第一页，增加阅读下面内容的兴趣。
  • 将说明的内容模块化，从大到小，从主到次的编排。

    想象一下，有一个空瓶子，你需要往瓶子里装大石块、粗砂砾和沙子。如何确保瓶子里能装进所有材料？当然，你会先放大石头，然后倒进砂砾，用沙子填满剩余的空间。 如果装材料时顺序弄错，就会失败。

标点

  • 中文语句的标点符号，均应该采取全角符号，这样可以与全角文字保持视觉的一致。
  • 如果整句为英文，则该句使用英文/半角标点。
